Security

Security.

Last updated: June 9 2026

FieldCheck's security posture, in plain language. Stack details, what we encrypt, and how to disclose vulnerabilities.

Hosting + transport

  • Frontend: Netlify CDN. HTTPS-only. HSTS enabled.
  • Backend worker: Cloudflare Workers. HTTPS-only. Cloudflare DDoS protection.
  • Data store: Cloudflare KV (for ephemeral cache) + D1 (for structured records). All TLS in transit.
  • Forms: Netlify Forms. Spam-filtered with Akismet. Encrypted at rest.

What's encrypted

  • All web traffic (TLS 1.2+).
  • All form submissions in transit and at rest.
  • Tier-jump alert subscriber emails (Netlify's encryption at rest).
  • D1 database (Cloudflare's managed encryption).

Access controls

  • Production worker + database access limited to single founder account.
  • 2FA enabled on Cloudflare, Netlify, and DNS provider accounts.
  • No third-party operational access.
  • Wrangler deploy keys rotated quarterly.

Application security

  • No user accounts on the public side means no credential database to compromise.
  • localStorage only for user state; no server-side session for visitors.
  • CORS restricted on worker endpoints to FieldCheck domains for state-changing endpoints.
  • Rate limiting on all worker endpoints via Cloudflare.
  • Photo cascade does not proxy user-supplied URLs (only worker-resolved athlete photos).

Vulnerability disclosure

If you discover a security vulnerability, please report it via the contact form on any verdict page with subject "Security" or directly. We commit to:

  • Acknowledgment within 24 hours.
  • Triage within 72 hours.
  • Public credit (with your permission) once the issue is patched.
  • No legal action against good-faith researchers following responsible disclosure.

What we don't do

  • We don't have a bug bounty program (yet). Disclosure is voluntary and credited.
  • We don't run third-party security analytics that collect visitor data.
  • We don't share our threat-detection logs.
