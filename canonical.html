Statistical Tests · Acceptance Criteria
  • 3.1 Paired t-test on per-player errors. For each Ri: H₁: E[error^baseline] > E[error^Ri]. Reject if p < 0.05 (one-sided).
  • 3.2 Holm-Bonferroni correction. Testing 6 refinements inflates Type I error. Test in order of expected effect size. Reject only those passing α / (k − rejected so far).
  • 3.3 Bootstrap CI on test/training ratio. Bootstrap 1000 random 70/30 splits. Acceptance: Mean ratio < 1.5 AND 95% CI upper bound < 2.0.
  • 3.4 Cumulative refinement test. Apply all approved refinements together. Target: Cumulative test RMSE < baseline test RMSE × 0.7 (30%+ reduction). If individual pass but cumulative fails → suspect interactions.
  • 3.5 Sport-specific subset tests. R4: NFL subset must improve significantly. R6: Soccer subset must not degrade.
  • 3.6 Tier-specific subset tests. R1-R4 targets bust prediction. Bust subset (n=8): current RMSE 1.18. Target with R1-R4: < 0.50.

Why this is a moat continuous learning loop
The data isn't the moat. The continuous refinement of the data is the moat. Each new athlete searched generates: source-success patterns (which fetchers fired) · school-anchor false-positives (training data for future validators) · composite variance (each call to Claude either confirms or stresses the grade) · downstream outcomes (the prediction ledger compares verdicts to what actually happened 6-24 months later). Every input strengthens future verdicts. The framework reuses every prior decision's evidence. Competitors building a D1-only product can't recreate the long-tail data layer because they don't have the validators or the volume. Six months of operation = a defensible position on tens of thousands of unknown amateurs that no one else even tries to grade.

Lessons logged · Tenets 28-29 candidates discipline gaps
  • Tenet 28 · Grep before naming · grep the PATTERN, not just the variable
    Refined May 23 evening after two consecutive sprints (V021.6, V021.7) shipped fixes to ONE site and missed the same broken pattern at a SECOND site. V021.6 tightened the D2D3_CONFERENCE_MAP lookup in fetchConferenceDataForPlayer (line 2955) but missed an identical loose-substring lookup in the D1-attractor block (line 5567). V021.7 had to ship as a second fix the same evening. Real rule: when fixing a pattern, grep for the pattern, not the variable. Five-second grep for .includes(...slice across worker.js would have caught both sites. Adding this to the pre-deploy checklist permanently.
  • Tenet 29 · "Acceptable noise" is the signal to STOP
    V021.1 famous-attractor rule explicitly carried the risk of breaking Donovan Dent — and I shipped it saying "acceptable noise." It wasn't. When reasoning explicitly identifies a working case that COULD break, that's not a signal to accept risk — it's a signal to narrow the fix. The V021.4.1 asymmetric rule (D1-attractor only when schoolHint is D2/D3) is the right shape: same goal, no Donovan Dent collateral damage.
  • Tenet 30 candidate · Pre-deploy data-structure audit
    V021.5 expanded D2D3_CONFERENCE_MAP from 30 to 116 entries and EXPOSED a latent loose-match bug that was harmless at 30 entries but broke Donovan Dent at 116. Adding "new mexico highlands" surfaced an existing flaw — the lookup code couldn't handle a hint that was a prefix of a mapped school. Rule: before expanding any lookup data structure, audit the lookup CODE against representative new keys. Mentally trace each new entry through the existing lookup once. 30 seconds of audit would have caught the regression before ship.
